# 실제 데이터 프로파일 — AI Hub 158.문학작품 낭송 (라벨링 데이터)

조사일 2026-06-17 / 대상: 사용자가 직접 반입한 라벨 데이터 2종
- Training: `1.학습레이블.json` (359MB) — 작품 33,401건 / 문장 230,898건
- Validation: `2.검증레이블.json` (44MB)
- ※ 음성(wav)은 미반입. **우리가 쓸 텍스트·라벨은 전량 확보됨** (음성 불필요).

## 1. JSON 구조 (작품 1건 = list의 1 element)
```
id                                      작품-낭송 단위 id (예: S0107-07-14)
voice {filename, sample_rate, duration, recored_at}   ← 음성 메타 (미사용)
reciter {gender, age}                    낭송자
recite_src
  ├ id
  ├ literature {id, genre, title, author, translator, publish_year, period}
  ├ text                                 작품(낭송분) 전체 원문
  ├ sentence_count, token_count
  └ styles [{style, emotion}]            작품 단위 대표 감정/스타일
sentences [                             ← 핵심: 문장 단위 배열
  ├ origin_text                          문장 원문 (★ 서비스가 쓸 문구)
  ├ voice_piece {filename, tr, ptr, duration}   음성/낭독 끊어읽기 (ptr=운율 단위)
  ├ styles [{style, emotion}]            문장 단위 감정/스타일 (1~2개)
  └ votes [{agree, voter{gender,age}}]   교차검증 투표 (대개 5인)
]
```

## 2. 라벨 체계 (★ 서비스 설계의 근간)
- **emotion = 7종 (대분류)**: 기쁨 / 슬픔 / 분노 / 불안 / 당황 / 상처 / 무감정
  - AI Hub 표준 6대 감정 + 무감정. → 두 축 설계의 **축②(정서 결)** 1차 분류로 직결.
- **style = 53종 (세부)**: 비통한·우울한·만족스러운·걱정스러운·두려운·괴로워하는·고립된·눈물이 나는·자신하는·신이 난… (데이터셋 소개의 "60감정 라벨"과 달리 **실제 고유값은 53종**.)
- **style ↔ emotion 관계**: 대체로 1:1이나 **13개 style은 2개 emotion에 걸침**(예: 노여워하는={분노,불안}, 염세적인={분노,슬픔}, 고립된={상처,당황}). → 깔끔한 계층이 아니라 **(style, emotion) 쌍 자체가 라벨**. ERD에서 style을 emotion 하위로 정규화하면 안 됨 → **(style,emotion) 페어를 라벨 엔티티로**.
- **문장당 라벨 수**: 1개(93,320) 또는 2개(137,578). **다중 라벨이 다수** → 문장↔라벨 **N:M**.

## 3. 데이터 품질 신호 (votes)
- 문장마다 대개 **5인 교차검증**. 표본(train 5천 작품) 기준:
  - 5/5 만장일치 29,634 · 4/5 = 4,430 · 3/5 = 184 · 2/5 = 11
- → **동의율 ≥ 4/5 문장만 후보 풀에 넣으면** 고신뢰 문구만 사용 가능. (품질 필터 칼럼으로 보존 가치 큼.)

## 4. 메타 분포
- **장르**: 희곡 12,067 / 소설 11,729 / 시 9,605
- **시대(period)**: 현대 9,191 / 근대 7,454 / 근대·현대 7,251 … 단 `'?','0.0','',  '미상','현대스페인'` 등 **더러운 값 존재** → 정제/표준화 필요.
- **literature**: 동일 작품(literature.id)이 여러 낭송분(recite_src)에 재등장 → 작품은 별도 엔티티로 정규화.

## 5. 서비스 설계에 주는 직접 함의
1. **이 데이터는 "출력 후보 풀"이다.** 사용자 입력의 부정도를 재는 자료가 아님. 문장들은 *문학 속 화자의 감정*으로 태깅돼 있음 → 우리의 3단계(라벨 매칭)·4단계(벡터 정렬)의 **재료**. 사용자 입력 분석(루브릭)은 여전히 하이브리드(KNU 사전+LLM)로 별도.
2. **핵심 단위는 문장(230,898).** 문장이 카드 문구가 된다. 작품/낭송분은 출처·맥락 메타.
3. **새 설계 결정 필요 — 매칭 방향**: 슬픈 사용자에게 *슬픈 문장(공감·동굴의 메아리)* vs *밝은 문장(전환·위로)* 중 무엇을 줄까? 데이터가 감정별로 갈려 있어 어느 쪽도 가능. → 사용자 확인 필요.
4. **ERD 윤곽(잠정)**: work(작품) 1—N recitation(낭송분) 1—N sentence(문장) N—M label(style,emotion). votes는 문장 품질 집계로 비정규화 저장(원천 투표 전량 보존은 과함). 음성 필드 전량 드롭.
