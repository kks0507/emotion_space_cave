## 1. 한 줄 소개

문학작품 낭송 데이터(AI Hub)를 활용해, 이용자가 털어놓은 심정을 감정 분석한 뒤
그에 맞는 문학 문구 1개를 골라 **디지털 카드 + 2문장 편지**로 돌려주는 서비스.
컨셉은 **"대나무 숲 / 동굴"** — 위로뿐 아니라 축하·격려·자랑 들어주기까지 포괄한다.

### 숨은 목적 (서비스의 진짜 존재 이유)
표면은 "위로 카드 서비스"이지만, 진짜 목적은 **학생 정서를 추적·관리해 자퇴·휴학 등 학업 중단을 예방하고
학생이 학적을 끝까지 유지하도록 돕는 것**이다.
→ 따라서 "위기"의 정의는 자살 위험 같은 임상적 위기 한 종류가 아니라,
  **정서적 이탈 → 학업 중단(드롭아웃) 위험**까지 포함한다.
→ 루브릭(§ D9 점수 → 행동 연결)은 이 학업중단 조기경보(early-warning) 관점에서 설계한다.
→ 이연된 관리자 모니터링(F3)의 궁극적 용도도 위기 학생 발견을 통한 **학적 유지 개입**이다.

### 핵심 파이프라인 (5단계)
1. 이용자 심정 입력
2. 감정 분석 — **긍/부가 아니라 "부정어가 얼마나 많은가"를 척도로** 부정도 측정
3. 감정 라벨과 매칭하여 문구 후보군 추출
4. RAG로 후보군 중 맥락에 가장 맞는 문장 **1개** 선별
5. 문구 강조 디지털 카드 생성 + 그 아래 문구를 활용한 **2문장 편지** 서술

## 7. ERD (D16 상세)

**원칙**: 최대한 정규화하되, *조회 성능에 직결되는 비정규화만* 명시적으로 허용.

```
work (작품 = literature)
  work_id PK (예: S0107)   title   author   translator
  genre (시/소설/희곡/시나리오)   period_raw   period_norm   publish_year

sentence (문장 = 카드 문구의 단위, 핵심)
  id PK   work_id FK→work
  recitation_src_id   seq          -- (비정규화) recitation 테이블 대신 그룹키+순서로 폴딩 → 원문 맥락 재구성용
  origin_text         ptr          -- ptr=운율 끊어읽기(낭독 데이터 특성, 보존)
  vote_total  vote_agree  agree_ratio  -- (비정규화) per-voter votes 원천은 드롭, 집계만. 품질 필터(≥0.8)에 매 조회 사용
  embedding F32_BLOB(dim)            -- 4단계 RAG용. 적재 후 별도 단계에서 채움(현재 nullable/미생성)

label (style·emotion 쌍 = 라벨 어휘)
  id PK   style   emotion   UNIQUE(style, emotion)   -- style이 emotion에 1:1로 안 떨어짐(13개 중복) → 쌍 자체가 라벨

sentence_label (N:M)
  sentence_id FK   label_id FK   PK(sentence_id, label_id)
```

**드롭한 것**: 음성 전 필드(voice/voice_piece/wav), 낭송자(reciter, 성우라 텍스트 서비스 무관), per-voter 투표 원천.
**비정규화 명시**: ① sentence.work_id(genre/period 조인 없이 필터), ② sentence.agree_ratio(품질 필터), ③ recitation을 sentence 컬럼으로 폴딩(테이블 절약).
**조회 경로**: 3단계=sentence_label⋈label에서 emotion 필터 + agree_ratio≥0.8 → 4단계=embedding 벡터 정렬. 메타(장르/시대)=work 조인.

### 8-1. ⚠️ Turso 대량 적재 교훈 (반드시 숙지 — 재적재 시 시간 좌우)
1. **원격(us-east-2) 왕복 지연이 병목.** libsql 클라이언트는 배치당 1회 네트워크 왕복 → 행 단위 `executemany`는 치명적으로 느림(행마다 왕복). **다중행 INSERT(한 statement에 N행)** 필수.
2. **배치 크기 실측 최적값**: sentence(9컬럼, 긴 한글 텍스트) = **1000행/배치(≈1.58s, 633행/s)**. 1500행은 페이로드 과대로 **정지**(에러 없이 멈춤). sentence_label(2컬럼, 정수) = 4000행 가능. 즉 **행수가 아니라 페이로드 크기가 한계** — 텍스트 컬럼은 배치를 키우지 말 것.
3. **Turso는 외래키(FK)를 강제한다** (일반 SQLite 기본 OFF와 다름!). → **적재 순서 고정**: work → label → **sentence(부모) → sentence_label(자식)**. 자식 먼저 넣으면 `SQLITE_CONSTRAINT: FOREIGN KEY constraint failed`.
4. **멱등 재개**: 모든 INSERT는 `INSERT OR IGNORE`. 중단 후 재개 시 DB에서 기존 id를 `SELECT`해 메모리 목록을 필터링하면 재전송 낭비 제거(finish_load.py 방식).
5. **libsql 파라미터는 tuple로** 전달(list 불가: `'list' object cannot be converted to 'PyTuple'`). 변수 한도 ~32766 내(1000행×9컬럼=9000 OK).
6. **백그라운드 stdout은 버퍼링** → 진행 확인은 별도 연결로 `SELECT count(*)` 직접 조회가 확실.
- 전체 규모: sentence 259,417 / sentence_label 414,048 → 최적 배치로 약 4~5분.
